# 数仓血缘与口径图谱系统需求文档

## 1. 文档目标

本文档用于定义“数仓血缘与口径图谱系统”的建设目标、业务范围、系统能力、数据模型、处理流程与验收标准，作为后续架构设计、开发实施和项目验收的依据。

## 2. 项目背景

当前数仓研发与治理场景中，通常存在以下问题：

- 表级血缘依赖人工维护，更新不及时，可信度低。
- 字段级血缘只能看到“字段到字段”的简单映射，缺少中间表达式与计算链路。
- 指标和字段口径散落在 SQL、任务脚本、表设计和人员经验中，难以统一沉淀。
- 业务方和研发方经常需要回答“某个字段怎么来的”“某张表做了什么业务处理”“某个指标依赖哪些字段”等问题，查询成本高。
- 随着任务数量增长，手工梳理上下游、口径与依赖关系的方式无法持续。

因此，需要建设一个基于 Spark 执行期采集的图谱系统，以实际任务执行结果为准，持续沉淀血缘与口径知识，形成可查询、可追溯、可问答、可自增长的数仓知识底座。

## 3. 建设目标

系统以 Spark `QueryExecutionListener` 为统一入口，在任务执行期采集 `QueryExecution` 可获得的多层信息，并逐步沉淀为数仓级图谱能力，达成以下目标：

- 构建可信的表级血缘图谱。
- 构建可追溯、可解释的字段级血缘图谱。
- 沉淀指标、字段、子查询、表的技术与业务口径。
- 为知识检索和大模型问答提供结构化知识底座。
- 基于任务持续运行实现图谱的自增长和逐步完善。

## 4. 建设范围

### 4.1 范围内

- Spark SQL / Spark 作业执行期采集。
- 面向离线数仓场景的表级与字段级血缘构建。
- SQL 计算逻辑、子查询逻辑、表级业务逻辑的抽取与存储。
- 图数据库或图存储模型的设计与写入。
- 基于图谱的检索查询能力。
- 面向知识库和大模型问答的知识供给接口。

### 4.2 暂不纳入

- 非 Spark 引擎任务的统一采集。
- 实时链路的完整统一治理。
- 自动生成高质量自然语言业务解释的最终效果保证。
- 跨组织、跨租户的权限体系细化设计。
- 完整的数据质量校验和修复闭环。

## 5. 术语定义

- 表级血缘：表与表之间的输入输出依赖关系。
- 一跳下游：从指定表直接产出的目标表，不递归展开。
- 字段级血缘：目标字段与来源字段及中间表达式之间的依赖关系。
- 表达式链路：字段从底层来源到最终产出的完整变换过程，包括函数、CASE WHEN、聚合、算术计算、别名映射等。
- 口径：字段、指标、表、子查询的计算逻辑与业务含义描述。
- 图谱：以节点和边形式组织的知识结构，可表达实体、属性与依赖关系。
- 任务：一次具体执行的 Spark 脚本或调度实例。

## 6. 总体建设思路

系统以“执行期采集优先、图谱沉淀驱动、逐步完善”为原则，整体分为四层：

1. 采集层  
   基于 Spark `QueryExecutionListener` 捕获作业执行后的逻辑计划、物理计划、输入输出信息、字段投影与表达式信息。

2. 解析层  
   解析 `QueryExecution` 中的计划树、表达式树、子查询结构、别名映射、聚合与函数依赖，抽取表、字段、任务、子查询、表达式等图谱元素。

3. 存储层  
   将任务执行结果写入图库或图模型存储，形成表级血缘、字段级血缘、表达式链路与口径信息，并通过任务增量更新持续完善。

4. 服务层  
   对外提供图谱查询、链路追溯、口径查询、知识检索及大模型知识供给接口。

## 7. 核心功能需求

### 7.1 表级血缘

系统需要支持以下表级血缘能力：

- 按表查看直接上游表。
- 按表查看直接下游表。
- 按表查看完整上下游链路。
- 支持指定表查看一跳下游。
- 支持按任务维度查看某次执行写入了哪些目标表、读取了哪些来源表。
- 支持区分“执行观察到的真实依赖”和“历史沉淀依赖”。

#### 7.1.1 表级血缘输出要求

- 可返回目标表、来源表、任务标识、执行时间、依赖类型。
- 可展示某张表是由哪些任务产出的。
- 可展示某张表作为输入被哪些任务消费。

### 7.2 字段级血缘

系统需要支持以下字段级血缘能力：

- 能从目标字段追溯到底层来源字段。
- 能展示中间表达式链路，而不是仅保留字段到字段的裸边。
- 支持字段别名、函数包装、算术运算、CASE WHEN、聚合、窗口函数等场景。
- 支持子查询跨层映射，不因中间层字段重命名而丢失链路。
- 支持字段血缘按任务版本或执行实例追溯。

#### 7.2.1 字段级血缘输出要求

- 返回目标字段到来源字段的完整依赖路径。
- 返回关键中间表达式节点。
- 返回字段所属表、子查询、任务上下文。
- 支持区分直接依赖字段和间接依赖字段。

### 7.3 指标与口径梳理

系统需要同时沉淀技术口径和业务口径。

#### 7.3.1 技术口径

- 描述指标或字段的计算逻辑。
- 记录表达式内容、聚合方式、过滤条件、分组维度、窗口定义等技术细节。
- 记录指标最终落在哪张表、哪个字段。

#### 7.3.2 业务口径

- 能描述子查询级业务逻辑，例如“筛选近 30 天活跃用户”“计算门店维度 GMV 聚合结果”。
- 能描述表级业务逻辑，例如“该表为门店日粒度销售汇总表”“该表用于宽表聚合供报表使用”。
- 支持任务级补充说明，用于沉淀脚本意图、业务背景、口径约束。

### 7.4 知识库与大模型问答支撑

系统作为大模型知识底座，需要支持以下典型问答：

- 某个字段怎么算出来的。
- 某张表的业务逻辑是什么。
- 某个指标依赖哪些字段。
- 某张表一跳下游有哪些。

#### 7.4.1 知识供给要求

- 图谱中每类核心实体需要具备结构化属性，便于检索与召回。
- 支持以图查询结果和结构化摘要形式输出给大模型。
- 支持将“表说明”“字段说明”“子查询说明”“任务说明”“表达式链路摘要”作为知识片段输出。
- 支持后续扩展为检索增强生成（RAG）场景。

### 7.5 图谱自增长与自完善

实际场景中，一张目标表通常由一个脚本任务产出。系统需要在任务持续执行过程中自动更新图谱：

- 每次任务执行后，采集输入表、输出表、字段映射并写入图库。
- 当同一目标表被重复执行时，支持版本覆盖或执行实例沉淀。
- 随着更多上下游任务逐步执行，图谱应自动补齐完整链路。
- 对于初次无法完全解析的链路，支持后续任务执行时继续补全。
- 支持历史任务回放导入，提升初始建图速度。

## 8. 核心实体与图模型

### 8.1 核心节点

- `Task`：任务节点，表示一次脚本或调度执行单元。
- `Execution`：执行节点，表示某次实际运行实例。
- `Table`：表节点。
- `Column`：字段节点。
- `Subquery`：子查询节点。
- `Expression`：表达式节点。
- `Metric`：指标节点。
- `Knowledge`：知识说明节点，用于沉淀业务解释、口径说明、备注信息。

### 8.2 核心边

- `READS_FROM`：任务 / 执行读取来源表。
- `WRITES_TO`：任务 / 执行写入目标表。
- `HAS_COLUMN`：表包含字段。
- `DERIVES_TO`：字段或表达式导向目标字段。
- `CONTAINS_SUBQUERY`：任务或查询包含子查询。
- `USES_EXPRESSION`：字段或子查询使用表达式。
- `BELONGS_TO`：字段属于表或子查询。
- `DESCRIBES`：知识节点描述目标实体。
- `DEPENDS_ON`：指标依赖字段、表或子查询。

### 8.3 关键属性建议

#### `Task`

- `task_id`
- `task_name`
- `script_path`
- `owner`
- `scheduler_type`

#### `Execution`

- `execution_id`
- `task_id`
- `run_date`
- `start_time`
- `end_time`
- `status`
- `spark_app_id`

#### `Table`

- `table_name`
- `db_name`
- `table_level`
- `business_domain`
- `table_comment`

#### `Column`

- `column_name`
- `data_type`
- `column_comment`
- `is_metric`

#### `Expression`

- `expression_id`
- `expression_text`
- `expression_type`
- `normalized_expression`

#### `Knowledge`

- `knowledge_type`
- `content`
- `source_type`
- `source_ref`
- `confidence`

## 9. 处理流程要求

### 9.1 采集流程

1. Spark SQL 或作业执行完成。
2. `QueryExecutionListener` 捕获 `QueryExecution` 上下文。
3. 提取任务标识、执行标识、输入表、输出表、逻辑计划、表达式树等信息。
4. 交由解析模块构建表、字段、表达式、子查询之间的依赖关系。
5. 将解析结果标准化并写入图存储。
6. 更新任务与执行实例的索引信息。

### 9.2 解析要求

- 优先基于逻辑计划进行血缘解析，避免仅依赖物理计划导致语义缺失。
- 对 `Project`、`Aggregate`、`Join`、`Filter`、`SubqueryAlias`、`Window` 等常见节点做专项处理。
- 支持表达式标准化，降低因格式差异造成的重复节点。
- 支持字段和表的限定名归一化。
- 支持任务维度与执行维度的双层建模。

### 9.3 入库要求

- 支持幂等写入，避免同一次执行重复入库。
- 支持按执行实例保留快照。
- 支持按任务主视角聚合最新状态。
- 支持节点和边的增量更新。

## 10. 查询与服务要求

### 10.1 图谱查询接口

至少提供以下查询能力：

- 查询某张表的直接上游。
- 查询某张表的直接下游。
- 查询某张表的一跳下游。
- 查询某个字段的完整来源链路。
- 查询某个指标依赖的字段集合。
- 查询某张表的业务逻辑说明。
- 查询某次任务执行的输入输出和字段映射摘要。

### 10.2 问答支撑接口

至少提供以下能力：

- 根据表名、字段名、指标名进行实体检索。
- 返回结构化图谱结果。
- 生成适合大模型使用的上下文摘要。
- 支持链路结果裁剪，避免返回过长上下文。

## 11. 非功能性要求

### 11.1 准确性

- 表级血缘准确率应优先保障。
- 字段级血缘允许分阶段迭代，但需要保留“可解释链路”。
- 每条链路需要可追溯到任务或执行实例来源。

### 11.2 可扩展性

- 支持后续扩展更多 Spark 逻辑节点解析规则。
- 支持后续接入更多计算引擎。
- 支持图谱模型扩展到指标、维度、标签等实体。

### 11.3 性能

- 单次任务执行后的采集与入库不能显著影响 Spark 主流程。
- 查询接口应支持常用血缘查询的秒级响应。

### 11.4 可运维性

- 需要具备采集成功率、解析成功率、入库成功率监控。
- 需要具备异常任务告警能力。
- 需要保留关键解析日志，便于问题排查。

## 12. 版本建设建议

### 12.1 第一阶段

目标：先建立可信的表级血缘和基础字段映射能力。

- 采集 Spark 执行输入表、输出表。
- 建立表级上下游查询。
- 建立基础字段级映射。
- 支持一跳下游查询。

### 12.2 第二阶段

目标：增强字段级血缘可解释性。

- 引入表达式节点。
- 支持子查询级链路保留。
- 支持聚合、函数、过滤等逻辑的解析与展示。
- 沉淀字段技术口径。

### 12.3 第三阶段

目标：建设业务口径与知识问答能力。

- 沉淀表级、子查询级业务说明。
- 建立指标实体与依赖关系。
- 输出适合大模型消费的知识摘要接口。

### 12.4 第四阶段

目标：实现图谱自增长与治理闭环。

- 支持历史任务导入与补图。
- 支持图谱版本治理。
- 支持解析缺陷回补和人工校正机制。

## 13. 验收标准

### 13.1 表级血缘验收

- 输入目标表名，可返回直接上游表列表。
- 输入目标表名，可返回直接下游表列表。
- 输入目标表名，可返回一跳下游结果。
- 每条依赖关系可追溯到至少一个任务或执行实例。

### 13.2 字段级血缘验收

- 输入目标字段，可追溯到底层来源字段。
- 链路中能看到关键中间表达式。
- 能正确处理常见字段别名和子查询映射。

### 13.3 口径验收

- 对核心指标或字段，能够展示技术计算逻辑。
- 对核心表，能够展示业务逻辑说明。
- 对典型子查询，能够沉淀子查询级业务解释。

### 13.4 问答验收

以下问题可返回结构化结果或明确答案：

- 某个字段怎么算出来的。
- 某张表的业务逻辑是什么。
- 某个指标依赖哪些字段。
- 某张表一跳下游有哪些。

## 14. 风险与难点

- Spark 计划解析复杂，字段级血缘存在实现难度。
- 同一逻辑在不同 SQL 写法下存在结构差异，需要做表达式归一化。
- 执行期能获取技术链路，但业务语义仍可能需要人工补充。
- 同一目标表可能被多个任务或多版本脚本写入，需要处理冲突与版本治理。
- 初期图谱完整度依赖任务覆盖率，需要通过持续运行与历史回放逐步补全。
